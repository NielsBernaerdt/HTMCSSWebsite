WitchCraft

2D Multiplayer Unity MVP

Description:

For this personal project, the main goal was to figure out realtime multiplayer.
I did this with a prototype where I explored the idea of "SpellCrafting" as main game mechanic.
During the process, My attention was drawn to finding a way of evading the obnoxious spaghetti animation graph. Here, Crossfade presented an excellent solution.
Furthermore, the opportunity presented itself for me to familiarize myself more with Unity's Input System and the alternative approaches to it rather than just connecting everything in the Player Input Component in the Unity Editor.


Features:

- Multiplayer
- Spell Crafting
- Code-Based Animations
- Input via Callbacks
- State Pattern

Multiplayer

In my first ever attempt at tackling multiplayer I used Netcode for GameObjects, a framework provided by Unity.
In this project emphasis was placed on creating my own network objects/variables, serializing my own data and implementing correct client/server RPC logic.
Minimal Quickjoin matchmaking functionality makes the client join any existing lobby or will create a new lobby if none are found.

Spell Crafting

A pawn with the wizard component can equip multiple spells. Casting these spells consecutively creates a new combined spell with unique effects, defined by the order in which the spells were cast.

Code-Based Animations

Animations are managed entirely within code, evading the spaghetti unity animator.
Combine this with each state (see State Pattern), which returns a hashed animation string to its corresponding animation, and optimal efficiency has been achieved.

Input Using Callbacks

To familiarize myself more with Unity's input system, I took a profound look at the documentation and came up with a code-only implementation using input actions and callbacks.

However, the requirement for hardcoded name values is a drawback, but I found this to be a more clean solution compared to relying on a correctly set up input component.

State Pattern

A state pattern is utilized by Pawns to transition between Idle, Move & Cast. This allows for efficient and organized management of game behaviors and provides a great opportunity to decouple a lot of the code.
